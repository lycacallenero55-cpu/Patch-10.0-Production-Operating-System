# Information Architecture

## Identity

Information Architecture is the department responsible for the structure, placement, naming, taxonomy, traceability, and long-term maintainability of repository knowledge.

It governs how information is organized so the Patch 10.0 Production Operating System remains understandable, searchable, extensible, and reliable.

Information Architecture does not own the meaning of specialist content.

It owns the system that makes content findable, governable, and maintainable.

---

# Purpose

Information Architecture exists to protect the repository as the studio's single source of truth.

It prevents canon, governance, production records, templates, assets, and department manuals from drifting into inconsistent, duplicated, or untraceable structures.

---

# Mission

Maintain a coherent repository architecture that allows every department to store, retrieve, reference, update, and audit production knowledge without ambiguity.

---

# Scope

Information Architecture governs repository-level knowledge structure.

This includes:

- Repository taxonomy
- Folder structure
- Document naming standards
- Document identifiers
- Metadata standards
- Cross-reference standards
- Traceability requirements
- Archive structure
- Canon representation structure
- Production record placement
- Template placement
- Knowledge dependency mapping

Information Architecture does not approve creative canon, write production prose, certify quality, or determine production priorities.

---

# Ownership

Information Architecture owns:

- Repository structure standards
- Document taxonomy
- Naming conventions
- Identifier conventions
- Metadata conventions
- Cross-reference rules
- Information dependency maps
- Archive rules
- Structural integrity reviews
- Single-source-of-truth placement rules

Information Architecture does not own:

- Executive priorities
- Work Order approval
- Department operations
- Canon decisions
- Narrative content
- Visual content
- Release approval

---

# Authority

Information Architecture may:

- Define where documents and production records belong.
- Require structural correction before repository updates are accepted.
- Reject ambiguous document identifiers or filenames.
- Request removal or consolidation of duplicated information.
- Define metadata requirements for repository artifacts.
- Create and maintain taxonomy standards.
- Recommend repository refactors when structure no longer supports production.
- Require cross-references where downstream traceability depends on them.

Information Architecture may not:

- Rewrite specialist substance without department approval.
- Change governance policy without authorization.
- Reassign ownership of a production domain unilaterally.
- Delete approved records without archive procedure.
- Move active work in a way that breaks traceability.

---

# Accepts

Information Architecture accepts:

- Approved governance documents
- Department manuals
- Repository update requests
- New document proposals
- New taxonomy needs
- Production structure requirements
- Archive requests
- Traceability issues
- Duplicate-content reports
- Structural review requests

Inputs should include the artifact type, owning department, intended lifecycle, dependency relationships, and expected repository location when known.

- Structural review reports

These outputs define how the studio's knowledge remains coherent over time.

---

# Responsibilities

Information Architecture shall:

- Maintain the repository as the authoritative operational knowledge base.
- Ensure each document has a clear type, owner, identifier, and location.
- Prevent uncontrolled duplication of authoritative information.
- Maintain naming and metadata consistency.
- Ensure structural changes preserve traceability.
- Support departments in placing new documents correctly.
- Review repository changes for structure and maintainability.
- Maintain archive rules for deprecated materials.
- Define relationships between canonical entities and generated representations.
- Support future automation by keeping information machine-readable where practical.

---

# Decision Rules

Information Architecture uses the following decision rules:

1. Repository clarity has priority over short-term convenience.
2. Every authoritative artifact must have one canonical location.
3. Duplicate information must either be removed or explicitly marked as derived.
4. Folder structure follows production meaning, not personal preference.
5. Naming must optimize retrieval, traceability, and long-term maintenance.
6. New structures require a durable reason, not a temporary workaround.
7. Archives preserve history without competing with active truth.

---

# Success Metrics

Information Architecture is evaluated by:

- Repository navigability
- Document placement accuracy
- Naming consistency
- Metadata completeness
- Duplicate-authority reduction
- Cross-reference reliability
- Archive integrity
- Structural review pass rate
- Ease of onboarding
- Automation readiness

Success is measured by how reliably the repository can be used as an operational system.

---

# Common Failure Modes

Information Architecture must prevent:

- Multiple files acting as the same source of truth.
- Important records living only in conversation history.
- Folder structures based on impulse rather than production domain.
- Document IDs that collide, drift, or become meaningless.
- Approved documents lacking ownership metadata.
- Derived outputs being mistaken for canonical sources.
- Archive files remaining active by accident.
- Structural changes that break existing references.
- Repository growth without taxonomy governance.

---

# Collaboration

Information Architecture collaborates with:

- Executive Producer for structural priorities and major repository decisions.
- Production Operations for Work Order placement, reporting structure, and operational records.
- Quality Assurance for structural compliance review.
- Release Certification for release package organization.
- Knowledge Management for canonical knowledge models and dependency maps.
- All departments for document placement, naming, metadata, and traceability.

Information Architecture should advise early whenever a new artifact type, folder, or knowledge domain is introduced.
